## Problem Statement

Korean Last Epoch players have no tool to build regex patterns for the in-game stash search using Korean stat names. The English tools (lastepoch.re, checkmatez/eternity) output English regex, which doesn't match Korean tooltips. Players must manually type Korean regex by guessing stat names or copying from wikis — painful with 1,112 affixes.

## What Makes This Cool

This is the missing piece for the Korean Last Epoch community. The community translation patch (fnrkp089/lastepoch-kr) lets players play in Korean, but stash search requires regex against Korean tooltip text. No Korean regex builder exists. This tool closes that gap — browse 1,112 affixes in Korean, click to build regex, paste into game. The bilingual toggle (KO/EN/KO+EN) is a unique feature that helps Korean players cross-reference English guides and wikis.

## Constraints

- Pure HTML + CSS + Vanilla JS — no framework, no build step for the web app
- Deploy to GitHub Pages (fnrkp089.github.io) — static only, no backend
- Data from lastepochtools.com extracted once via browser console, stored as committed JSON
- Build script (build_data.mjs) runs with Bun to produce affix_data.json
- Game aesthetic: dark theme (#0f0d0a), gold accents (#d4b870), tier-colored badges
- Fonts: Noto Sans KR (Korean), JetBrains Mono (regex/code)
- Mobile-first, max-width 800px centered

## Premises

1. The Korean translation patch uses the same stat names as lastepochtools.com's ko.json — regex built from that i18n data will match in-game tooltips. **AGREED**
2. One-time manual data extraction from browser console is the right approach. Re-run when game patches change data. **AGREED**
3. Playground section (test regex against sample items) is v2. Core value is affix browser + regex output. **AGREED**
4. GitHub Pages (static, no backend) is sufficient — no user accounts or server-side needed. **AGREED**

## Approaches Considered

### Approach A: Faithful Clone
Mirror lastepoch.re exactly in Korean. Single index.html + style.css + app.js. All 1,112 affixes as collapsible cards. Search, filter, click-to-add.

- **Effort:** S | **Risk:** Low
- **Pros:** Proven UX, simplest to build, easiest to maintain
- **Cons:** No differentiation beyond language, doesn't improve on reference tool's pain points

### Approach B: Enhanced Korean-First Experience (CHOSEN)
Same core (affix browser + regex output) with enhancements: categorized affix tabs (Core/Offensive/Defensive/etc.), inline tier-range regex preview before clicking, bilingual toggle (KO/EN/KO+EN side-by-side), and macro reference integrated as collapsible panel. Still pure HTML/CSS/JS, still GitHub Pages.

- **Effort:** M | **Risk:** Low-Med
- **Pros:** Category tabs reduce cognitive load, bilingual toggle helps cross-reference EN guides, tier preview shows regex before committing, macro reference inline reduces tab-switching
- **Cons:** More code, category grouping depends on displayCategoryKeys data quality

### Approach C: Regex Recipe Builder
Users describe what they want ("T7 crit gear for Rogue") and get pre-built regex recipes. Curated library + custom builder.

- **Effort:** L | **Risk:** Med-High
- **Pros:** Most beginner-friendly, unique angle, could go viral
- **Cons:** Curating recipes is manual ongoing work, doesn't help advanced users, natural language matching fragile without backend

## Recommended Approach

**Approach B: Enhanced Korean-First Experience.** The enhancements (category tabs, bilingual toggle, tier preview) add meaningful value for Korean players who constantly cross-reference English resources. The wireframe was approved — see `sketch.html` in the project root for the visual reference.

### Key UI Sections

1. **Header** — Title with bilingual toggle (한국어 / EN / KO+EN)
2. **Output Bar** — Always-visible regex display, mode toggle (Regex/Expression), copy button, character count
3. **Search + Filters** — Korean text search (matches name_ko, title_ko, filter_ko, name_en), filter dropdowns for type/equipment/class
4. **Category Tabs** — 전체, 핵심, 공격적인, 방어적인, 상태이상, 클래스 특정, 신상, 기타
5. **Affix Cards** — Collapsed: T1 range + Korean name + English subtitle + meta (title, type badge, equipment list). Expanded: all 7 tiers with color-coded badges and inline regex preview
6. **Regex Examples** — 5 hardcoded static Korean regex patterns with descriptions (no curation path, no dynamic generation — just static strings in `app.js`):
   - `/(2[6-9]|30)%.*?이동/` → T7 이동 속도(26-30%) 찾기
   - `/[0-9]+%.*?(냉기|화염|번개) 저항/` → 원소 저항 아이템 찾기
   - `/(2[1-6])%.*?체력/` → T7 체력(21-26%) 찾기
   - `/치명타/` → 치명타 관련 아이템 찾기
   - `T7&/방어력/` → T7 방어력 접사 아이템 찾기
7. **Macro Reference** — Read-only collapsible table of game macros (LP, WW, T7, FP, etc.) with Korean labels. Purely informational, no interaction.

### Bilingual Toggle Behavior

| Mode | Card title | Card subtitle | Regex output |
|------|-----------|---------------|-------------|
| 한국어 | Korean stat name | Hidden | Korean (`/치명타 확률/`) |
| EN | English stat name | Hidden | Korean (always — regex must match Korean tooltips) |

Regex output is **always Korean** regardless of display mode — the toggle only changes what the user sees on cards to help cross-reference. The regex must match in-game Korean tooltips.

### Regex Generation Logic

- **Simple click (affix name):** Inserts `/한국어스탯이름/` into output bar. Uses `filter_ko` (FilterOverride) if non-empty, otherwise `name_ko` (DisplayName).
- **Tier click:** Generates range-aware regex combining the tier's numeric range with the Korean stat name. Example: T7 Armor with range 148–156 → `/(14[8-9]|15[0-6]).*?방어력/`.

#### Tier Range Regex Algorithm

Given `min` and `max` integers, generate a regex that matches any integer in `[min, max]`:

1. If `min === max`: output the literal number (e.g., `42`).
2. If `min` and `max` share the same prefix digits and differ only in the last digit: output `prefix[lastMin-lastMax]` (e.g., 148–149 → `14[8-9]`).
3. Otherwise: decompose into sub-ranges that each share a prefix. Union them with `|` and wrap in `(...)`. Strategy:
   - Split at decade/century boundaries. E.g., 148–156 splits into 148–149 (`14[8-9]`) and 150–156 (`15[0-6]`).
   - For ranges crossing hundreds (e.g., 98–102): 98–99 (`9[8-9]`), 100 (`100`), 101–102 (`10[1-2]`).
4. For percentage values, append `%` after the number pattern.
5. Full pattern: `/(rangePattern).*?statName/` where `.*?` bridges any text between number and stat name.

A reference implementation of integer-range-to-regex exists as the npm package `to-regex-range`. For this project, a simple recursive decomposition in `app.js` (~30 lines) is sufficient given the typical game value ranges (0–500).
- **Multiple selections:** Combined with `|` operator: `/스탯1|스탯2/`
- **Regex mode vs Expression mode:** Regex mode outputs a bare `/pattern/` string. Expression mode adds a text input field before the regex where the user types macro expressions (LP3+, T7, 2T6+, etc.) manually. The macro text is prepended with `&` to the regex. Example: user types `T7` in the macro field → output becomes `T7&/치명타 확률/`. The macro reference table helps users know which macros exist but has no click interaction — macros are always typed.

### Affix Record Schema (`affix_data.json`)

```json
{
  "id": 5,
  "name_ko": "치명타 확률",
  "name_en": "Critical Strike Chance",
  "title_ko": "암살자의",
  "title_en": "Assassin's",
  "filter_ko": "",
  "filter_en": "",
  "type": "prefix",
  "category": "공격적인",
  "multi": false,
  "class_ko": "전체",
  "class_en": "All",
  "equip_ko": ["한손 도끼", "단검"],
  "equip_en": ["One-Handed Axe", "Dagger"],
  "tiers": [
    { "tier": 1, "rolls": [{ "min": 15, "max": 20, "suffix": "%" }] },
    { "tier": 7, "rolls": [{ "min": 45, "max": 55, "suffix": "%" }] }
  ]
}
```

- `filter_ko`: The FilterOverride value. Non-empty when the in-game search text differs from the stat display name. Use this for regex output when non-empty.
- `multi`: `true` for multiAffixes (multiple stat rolls per tier), `false` for singleAffixes.
- `rolls[].suffix`: `"%"` for percentage values, `""` for flat values. The build script converts raw multipliers (0.15 → 15) and marks the suffix. Flat values (e.g., +58 armor) are kept as-is.
- `category`: Assigned from `displayCategoryKeys` in the affix structure data. `displayCategoryKeys` is an array of i18n key strings (e.g., `["Item_Affixes.Affix_Category_Core", "Item_Affixes.Affix_Category_Offensive", ...]`) that defines the category ordering. Each affix's category is determined by its position in `categoryHeaders` (a map of affix IDs to category indices). The build script resolves the i18n key to the Korean category name. If an affix has no category mapping, it defaults to `"기타"` (Other). If an affix appears in multiple categories, use the first.

### MultiAffix Behavior

MultiAffixes (534 entries, ~48% of total) have multiple stat rolls per tier. Display behavior:

- **Card collapsed:** Show all stat names stacked, e.g.:
  ```
  15%-20% 치명타 확률 증가
  10-15 민첩
  ```
- **Card expanded:** Each tier shows all stats for that tier, stacked per roll.
- **Click behavior:** Clicking the card adds a regex for the **first** stat name (the primary stat). Clicking a specific stat line within the card adds that stat's regex.
- **Tier click for multiAffix:** Each stat line within a tier is independently clickable and generates a regex using that stat's own range and name. Clicking the tier badge itself (T1, T2, etc.) uses the first stat's range + name as default.

### Critical Implementation Details

1. Regex output is KOREAN (matches Korean tooltips in-game)
2. Macros are ENGLISH (LP, T7, WW — engine keywords regardless of language)
3. `filter_ko` takes priority over `name_ko` for regex output when non-empty
4. Tier range values: build script converts raw multipliers to display values (0.15 → 15, suffix "%"). Flat values kept as-is (suffix ""). The `suffix` field in each roll signals the type.
5. ~1,112 affixes — cards rendered lazily per category tab. On initial load, only the active tab's cards are in the DOM. Switching tabs renders that tab's cards on first visit and caches them. This keeps initial DOM size manageable for the 2s mobile target.
6. **Tooltip format assumption:** Korean tooltips display as `{value}{suffix} {stat_name}` (e.g., `45% 치명타 확률 증가`). Regex pattern is `/(range_pattern).*?스탯이름/`. This assumption MUST be verified against actual in-game tooltips in Step 5 of Next Steps.
7. **Empty states:** Zero search results shows "검색 결과 없음" message. If `affix_data.json` fails to load, show an error banner with reload prompt.
8. **Build script validation:** `build_data.mjs` asserts expected field counts (578±20 singleAffixes, 534±20 multiAffixes) and throws if counts fall outside the tolerance. This catches structural changes in game data.
9. **Estimated `affix_data.json` size:** ~300-500KB uncompressed. GitHub Pages serves with gzip, so transfer size should be ~50-100KB. Well within the 2s mobile target.
10. **Font loading strategy:** Load Noto Sans KR with `font-display: swap` and preconnect to Google Fonts CDN. Use only weights 400 and 700. JetBrains Mono for regex display only (weight 400). This prevents font-blocking on initial render.
11. **Bun dependency:** Minimum Bun v1.0+. The build script uses only `fs` and `JSON` — no Bun-specific APIs. Running with `node build_data.mjs` should also work as a fallback.
